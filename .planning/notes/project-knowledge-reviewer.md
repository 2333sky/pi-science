# 项目知识文档与 Reviewer Agent 设计构想

## 1. 产品定义

为每个 Pi-Science 项目维护一套持续演化的项目知识系统。它不是聊天记录的简单摘要，而是项目当前最可信、最有用状态的集合。

每次对话、研究任务或文件产出都可能产生新的知识。独立的 Reviewer Agent 负责识别这些知识增量，整理成待确认修改；只有用户确认后，修改才能进入正式项目文档或执行真实文件操作。

核心原则：

1. 正式项目文档是经过用户确认的可信事实源。
2. Reviewer Agent 只能提出建议，不能直接修改正式知识或移动真实文件。
3. 不依赖“对话结束”判断。Reviewer 持续处理新增事件，并把候选内容静默放入待整理区。
4. 所有结论都应尽可能关联来源会话、消息、文件、运行记录和外部引用。
5. 知识整理与文件整理共用审批机制，但必须区分风险等级。

## 2. 用户价值

当前项目知识通常散落在聊天记录、研究报告、代码、数据和各类 Office 文件中。随着会话数量增加，用户会逐渐遇到：

- 找不到此前得到的重要结论；
- 不清楚某个决策为什么作出；
- 新会话不了解项目当前状态；
- 新研究与旧结论冲突但没有提醒；
- 文件命名和目录结构逐渐混乱；
- 为生成报告、PPT 或项目总结而重复整理信息。

项目知识系统应让用户随时回答：项目要做什么、已经知道什么、依据是什么、决定了什么、还有什么没有解决、下一步应该做什么。

## 3. 总体信息流

```text
对话 / 研究 / 文件 / 实验运行
                │
                ▼
         Reviewer Agent
                │
     识别增量、去重、分类、审查
     关联证据、检测冲突、评估风险
                │
                ▼
             待整理区
        ┌───────┴────────┐
        ▼                ▼
     知识修改          文件操作
        │                │
        └───────┬────────┘
                ▼
      用户接受 / 编辑 / 拒绝
                │
                ▼
  项目文档、知识索引、文件系统、历史记录
```

Reviewer 不需要判断一场对话是否已经结束。它可以通过最后处理的消息或事件游标增量运行，将结果放入待整理区，并只在形成关键结论、发现冲突或积累到合理数量时轻量提醒用户。

## 4. 三层知识结构

### 4.1 项目概览

项目根目录新增用户可见的 `PROJECT.md`，作为简洁、稳定、适合注入新会话的项目快照：

- 项目目标与范围；
- 当前状态；
- 核心结论；
- 关键决策；
- 约束与风险；
- 开放问题；
- 下一步。

`PROJECT.md` 不保存全部研究细节，建议保持在可以快速阅读和作为模型上下文使用的长度。

现有 `KNOWLEDGE.md` 当前由 `harness/` 初始化，主要包含通用科学工具和工作流说明。第一版不应直接将其改造成项目文档，以免混淆“Agent 通用知识”和“项目特定事实”。后续可以重新明确两者的命名及边界。

### 4.2 结构化知识库

详细结论和证据保存在项目的 `.pi-science/knowledge/` 中，包含：

- findings：事实和研究发现；
- conclusions：基于证据形成的结论；
- decisions：项目决策及理由；
- hypotheses：尚未验证的假设；
- questions：开放问题；
- tasks：明确的后续行动；
- glossary：项目术语；
- timeline：重要事件与状态变化。

这些知识条目是 `PROJECT.md` 的证据层和历史层。正式条目必须带来源，且能够表达“被新条目替代”“存在冲突”“已经过期”等状态。

### 4.3 原始记录

会话、研究原文、运行日志、文件版本和外部引用继续保留在已有系统中，不重复复制进项目文档。知识条目通过稳定标识符关联它们。

## 5. Reviewer Agent

### 5.1 主要职责

Reviewer Agent 应识别“知识增量”，而不是生成普通对话摘要：

- 新事实：新发现或新获取的数据；
- 新结论：由已有证据支持的判断；
- 新决策：选择了什么、为什么以及替代方案；
- 新假设：值得保留但尚未验证的推测；
- 新问题：需要进一步研究或验证的内容；
- 新行动项：具体下一步；
- 项目变化：目标、范围、约束或状态的变化；
- 新产物：重要文件、数据、代码或报告；
- 冲突和失效：新信息是否与正式知识矛盾或使其过期。

Reviewer 还负责：

- 合并重复候选内容；
- 区分事实、推断、假设和决策；
- 评估相关性、可信度、新颖性、稳定性和可执行性；
- 为每条候选修改附加来源和理由；
- 根据项目已有的整理规则提出文件移动、重命名和新建目录建议。

### 5.2 明确边界

Reviewer Agent 不得：

- 未经确认直接修改 `PROJECT.md`；
- 未经确认把候选内容写成正式知识；
- 未经确认移动或重命名用户文件；
- 把没有来源的模型推断标记为事实；
- 自动删除真实文件；
- 因为单次对话措辞变化就反复改写稳定结论；
- 读取工作区外的路径或建议把文件移出工作区。

### 5.3 候选知识结构

示例：

```yaml
id: proposal-20260715-001
type: conclusion
title: Custom API 已可用于对话
summary: 自定义 API 自动发现的模型现在可以在输入框中选择并用于新消息。
confidence: high
status: proposed
importance: important
source:
  session_id: session-32
  message_ids: [128, 131]
related_files:
  - path: frontend/src/components/model-selector.tsx
    file_id: file-7b8d
supersedes:
  - knowledge-item-018
reason: 模型选择问题已经修复，并完成相关验证。
created_at: 2026-07-15T00:00:00+08:00
```

第一版可以使用 JSON 或 JSONL 保存候选项；最终格式应在实现前通过一次数据模型设计确定。

## 6. Reviewer 的运行时机

不使用“对话结束”作为主要触发条件，采用事件驱动加批量整理：

1. 每条新消息、研究任务完成、运行结束或文件变更都产生可处理事件。
2. Reviewer 记录最后处理游标，只消费新增内容。
3. 普通候选静默进入待整理区，不打断对话。
4. 重复候选在后台合并，低价值内容被过滤。
5. 出现以下情况时进行轻量提醒：
   - 形成明确结论或决策；
   - 新信息与正式知识冲突；
   - 研究任务已完成；
   - 待确认内容积累到一定数量；
   - 用户准备生成报告、PPT 或项目总结。
6. 用户也可以随时主动点击“整理到项目文档”。

为了控制成本，Reviewer 可以先通过规则筛选明显事件，再由模型处理高价值片段；是否采用独立模型、低成本模型或当前对话模型，留待实现阶段决定。

## 7. 待整理区

建议在应用中增加统一的待整理入口：

```text
待整理
├── 知识更新
│   ├── 新结论
│   ├── 决策
│   ├── 假设
│   └── 开放问题
├── 文件整理
│   ├── 移动
│   ├── 重命名
│   └── 新建目录
└── 风险与冲突
    ├── 重复文件
    ├── 知识冲突
    └── 失效引用
```

每条建议支持：

- 查看修改前后差异；
- 查看 Reviewer 的理由；
- 跳转到来源会话、消息、文件或运行；
- 接受、编辑后接受或拒绝；
- 逐条处理或按组批量处理；
- 查看风险等级和受影响对象。

被拒绝的建议不进入正式知识，但可以保留精简的拒绝记录，用于降低相似建议的出现频率。

## 8. 文件分类框架

采用“基础真实目录 + 应用内逻辑视图 + AI 增量建议”的混合模式。

### 8.1 基础真实目录

```text
project/
├── PROJECT.md
├── sources/       # 文献、网页和外部参考资料
├── research/      # 研究过程、实验记录和分析结果
├── data/          # 原始数据与处理后数据
├── work/          # 草稿和中间产物
├── deliverables/  # 最终 Word、PPT、Excel、PDF 等
├── archive/       # 已过期但仍需保留的内容
└── .pi-science/
    ├── knowledge/
    ├── inbox/
    ├── history/
    ├── index.json
    └── policy.yaml
```

基础目录必须保持浅层、稳定。Reviewer 根据项目内容提出增量目录，例如 `research/literature-review/` 或 `data/processed/`，但不能自动创建。

### 8.2 应用内逻辑视图

真实文件位置保持唯一，应用根据元数据让同一个文件出现在多个视图中：

- 按文件类型；
- 按研究主题；
- 按研究任务；
- 按关联结论；
- 按时间；
- 按状态，例如待处理、有效、过期、存在冲突。

知识条目应优先通过稳定 `file_id` 关联文件，路径只是文件当前的位置。这样文件移动后，知识关系不会丢失。

### 8.3 增量调整约束

- 至少出现多个相关文件后再建议建立新目录；
- 默认目录深度不超过三级；
- 使用稳定研究主题，不使用临时聊天标题建目录；
- 用户可锁定目录，Reviewer 不再建议修改；
- 分类重叠时优先使用标签和逻辑视图；
- 多次被拒绝的建议应降低权重；
- 用户确认的目录和命名习惯写入 `.pi-science/policy.yaml`；
- Reviewer 可以提出整理规则，但规则本身也必须经过确认。

## 9. 文件移动与重命名审批

文件建议应展示完整操作计划，而不是只给自然语言描述：

```diff
+ mkdir data/experiments/culture-optimization/

→ data/result-final-v2.xlsx
  data/experiments/culture-optimization/results.xlsx

→ 文档/实验记录最终版.docx
  research/culture-optimization/experiment-notes.docx
```

执行前必须完成：

- 工作区路径安全检查；
- 目标路径重名检查；
- 文件是否被占用或发生新修改的检查；
- 项目文档和知识索引引用扫描；
- Git 工作区状态提示；
- 操作后受影响路径预览。

一组文件操作应作为事务执行：全部成功或全部回滚。执行结果写入 `.pi-science/history/`，并提供一键撤销。删除文件不包含在普通 Reviewer 建议中，应使用独立的高风险流程。

## 10. 与 Pi-Science 现有能力的衔接

当前项目已经具备以下基础：

- 工作区级 `.pi-science/` 数据目录；
- 工作区级会话保存与恢复；
- 文件浏览、读取、上传、删除和预览；
- 文件版本与会话来源谱系；
- 实验运行记录；
- Reviewer 类型的子 Agent 扩展能力；
- DOCX、XLSX、PPTX 等文档预览。

因此该功能可以在现有工作区模型上增量建设，不需要重新定义项目边界。最重要的新模块预计包括：

1. 项目知识及候选修改的数据模型；
2. Reviewer 事件队列和增量处理游标；
3. 待整理区 API 与界面；
4. `PROJECT.md` 的结构化补丁和版本历史；
5. 逻辑文件索引、稳定文件 ID 和项目整理策略；
6. 可预览、可回滚的文件移动与重命名事务。

这里仅记录未来实现触点，不代表本次已经授权修改上述代码。

## 11. 建议的 MVP 边界

第一版先验证“Reviewer 提议、用户确认、正式写入”的核心闭环：

1. 每个新项目创建 `PROJECT.md` 和 `.pi-science/inbox/`。
2. 用户主动触发“整理当前会话”，暂不做全自动后台运行。
3. Reviewer 只识别结论、决策、开放问题和下一步四类内容。
4. 待整理区展示来源、理由和 Markdown 差异。
5. 用户逐条或批量确认后更新 `PROJECT.md`。
6. 文件整理第一版只生成建议和预览，不立即执行真实移动。

在闭环可靠后，再增加异步 Reviewer、冲突检测、文件事务、逻辑多视图和项目整理偏好学习。

## 12. 风险与待定决策

进入实现前仍需确定：

- 正式知识的权威数据源是 Markdown、JSON/JSONL，还是两者组合；
- `PROJECT.md` 是直接编辑的主数据，还是由结构化知识渲染的项目快照；
- Reviewer 使用哪个模型，以及成本、频率和并发限制；
- Reviewer 如何安全处理外部文档中的提示注入内容；
- 候选知识的置信度是否允许用户调整；
- 多条候选同时修改同一章节时如何合并；
- 用户手动编辑 `PROJECT.md` 后如何与结构化知识同步；
- 文件稳定 ID 在重命名、移动、复制和 Git 操作中的生成及保持方式；
- 文件操作失败时的跨平台回滚策略。

## 13. 进入代码实现的触发点

本设计当前处于产品探索阶段，不立即实现代码。满足以下条件后，应明确提醒用户可以进入实现阶段：

1. 用户确认 MVP 是否采用“主动整理当前会话”，还是第一版就需要异步 Reviewer；
2. 确定 `PROJECT.md` 与结构化知识库的权威关系；
3. 确定待整理区的基本交互草图；
4. 确定第一版是否执行真实文件移动，或只提供文件操作建议；
5. 将 MVP 拆成后端数据层、Reviewer 流程和前端审批界面三个可验收阶段。

到达上述节点时，先向用户说明即将开始修改哪些模块、预计增加哪些数据文件和如何手动验收，再开始写代码。

## 14. 实现结果（2026-07-15）

本设计已完成首轮实现与验收：

- 工作区初始化会创建 `PROJECT.md`、基础真实目录和 `.pi-science` 知识结构；
- Reviewer 使用独立临时 Pi RPC 进程，兼容现有内置模型和 Custom API；
- 对话结束后可异步自动审阅，也可从输入框或项目知识页手动触发；
- 模型输出必须通过结构、来源消息、相关文件和路径安全校验；
- 待整理区支持查看差异、证据、理由、安全预览、编辑、接受、拒绝和批量处理；
- 接受知识建议后才更新结构化知识及 `PROJECT.md`；
- 文件建议支持 `mkdir`、`move`、`rename`，执行前检查路径、冲突、锁定目录、Git 状态和引用；
- 文件操作使用事务执行，失败自动回滚，成功后可从历史记录撤销；
- 文件索引使用稳定文件 ID，并提供按类型、主题和月份的逻辑视图；
- 项目文档每次更新生成可恢复版本；
- 整理规则保存到 `.pi-science/policy.yaml`，接受和拒绝结果用于累计项目偏好；
- 前端提供 Project Knowledge 页面、侧栏待确认徽标、暗色模式和窄屏布局。

对应 AI 行为契约见 `project-knowledge-reviewer-ai-spec.md`，验收证据见 `project-knowledge-reviewer-uat.md`。
